# Source Style Notes

These notes summarize observable writing habits from older ghack.dev posts provided as style references.

## Structure

- Opens with practical context: current work, a learning journey, a real product problem, or a concrete motivation.
- Introduces theory after the reader understands the real-world reason for caring.
- Frequently turns concepts into questions, then answers them section by section.
- Uses examples from familiar systems before abstracting: e-grocery search, image rendering, frontend components, Apple Developer Academy projects.
- Often ends by saying the topic is only the beginning and inviting the reader to continue following the next step.

## Sentence Rhythm

- Direct, explanatory sentences with occasional long chains of reasoning.
- Repeats important nouns rather than replacing them with too many pronouns.
- Uses connective phrases generously: "Because", "So", "However", "In this case", "Based on", "To be clear".
- Uses conversational honesty: "I think", "in my opinion", "so yeah", "actually", "luckily", "the first thing that pops up in my mind".

## Technical Explanation

- Starts top-down from user/business/problem perspective before implementation.
- Explains why a naive solution is not enough before introducing a better model.
- Uses tables, lists, and code blocks when they clarify the mental model.
- Keeps implementation examples close to the problem rather than abstract library tour.

## Mesmer Adaptation

- For Mesmer project posts, translate personal "I" habits into a project "we" voice.
- Keep the human curiosity and practical examples, but make the authorial stance "Mesmer AI" or the Mesmer team.
- Present research as something we test and operationalize, not something we merely summarize.
- End with GitHub as the place to inspect implementation details.
